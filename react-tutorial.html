Chapter 1

React Fundamentals

What is React?

React is a JavaScript library for building user interfaces, created by Facebook (Meta) in 2013. It is not a framework — it is a library that focuses specifically on the view layer of your application.

Think of it this way: if HTML and JavaScript are like building with individual bricks, React lets you build with pre-assembled walls and rooms called components.

💡 Real Project Example: Look at your src/pages/Home.jsx — the Home page is composed of three smaller pieces: <Header />, <HomeMainContent />, and <Footer />. Each is an independent, reusable component.

Why React is Used?

ProblemReact's Solution
Websites feel slow when data changesVirtual DOM — only updates what changed
Code gets messy as apps growComponents keep code organized
Repeating the same HTML is tediousReusable components — write once, use everywhere
Changing page requires full reloadSPA — instant navigation without page refresh
JavaScript DOM manipulation is verboseJSX — write HTML-like syntax in JavaScript

What Problem Does React Solve?

Before React: If a notification count changed from 3 to 4, you'd manually find the DOM element and update it:

// Pure JavaScript — imperative, tedious
const countElement = document.getElementById('notification-count')
countElement.textContent = '4'
countElement.style.display = 'block'

With React: You just change the data — React updates the UI automatically:

// React — declarative
const [count, setCount] = useState(3)
// Call setCount(4) and React updates the UI instantly

Difference Between HTML, JavaScript, and React

  • HTML is the structure — the skeleton of the page.
  • JavaScript is the behavior — the muscles that move things.
  • React is the blueprint — a system for organizing HTML + JavaScript together efficiently.
🍽️ Restaurant Analogy:
HTML = The menu printed on paper (static)
JavaScript = The waiter who changes what's listed based on availability (dynamic)
React = A digital menu board that automatically updates prices and availability without needing a new menu printed

Virtual DOM

The Virtual DOM is a lightweight JavaScript copy of the real DOM. When your HomeMainContent component re-renders, React:

  1. Takes a snapshot of the Virtual DOM (before change)
  2. Creates a new Virtual DOM (after change)
  3. Uses a diffing algorithm to find what changed
  4. Updates only those specific elements in the real DOM
// React Virtual DOM in action
function ServiceCard({ title, description }) {
  return (
    <div className="service-card">
      <h3>{title}</h3>
      <p>{description}</p>
    </div>
  )
}
// React only updates the parts that change — not the entire card!

SPA (Single Page Application)

Your project is an SPA. When you click "Services" in the Header, the browser does not reload the page and request a new HTML file from the server. Instead, React Router intercepts the click, updates the URL, and swaps out only the content section — all without a full page reload.

🔍 How your project demonstrates SPA: Open your browser's Network tab and navigate between Home, About, Services, Team, Contact, and Blog Details pages. You'll see that the page never fully reloads — only tiny data requests (if any) occur. The experience feels as fast as a desktop app.

Chapter 3

JSX Deep Dive

What is JSX?

JSX stands for JavaScript XML. It is a syntax extension that lets you write HTML-like code inside JavaScript. JSX is not HTML — it gets compiled to JavaScript function calls by Vite (using a tool called SWC or Babel).

⚡ Important: Browsers cannot read JSX! Vite's build process converts all JSX into plain JavaScript before the browser sees it.

Here's what happens behind the scenes:

// What YOU write in .jsx files:
<div className="container">
  <h1>Welcome to ITCO</h1>
</div>

// What Vite compiles it to:
React.createElement("div", { className: "container" },
  React.createElement("h1", null, "Welcome to ITCO")
)

// What the browser actually sees:
// <div class="container"><h1>Welcome to ITCO</h1></div>

How JSX Differs from HTML

JSX RuleHTMLJSX
class attributeclass="container"className="container"
for attribute<label for="email"><label htmlFor="email">
Self-closing tags<br><br />
JavaScript expressionsNot possible{variable} or {function()}
Multiple root elementsAllowedMust wrap in a fragment: <></>

Rules of JSX

⚠️ Rule 1: Return only one root element
A component MUST return a single root element. Use a <div> or <></> (Fragment) to wrap multiple elements. 
// ❌ Wrong — two root elements
function Header() {
  return (
    <nav>...</nav>
    <div>...</div>  // Syntax error!
  )
}

// ✅ Correct — wrapped in Fragment
function Header() {
  return (
    <>
      <nav>...</nav>
      <div>...</div>
    </>
  )
}
⚠️ Rule 2: Use className instead of class
class is a reserved keyword in JavaScript. React uses className instead. 
// ❌ Wrong
<div class="container">

// ✅ Correct
<div className="container">
⚠️ Rule 3: Close all tags — even self-closing ones 
// ❌ Wrong
<img src="logo.png">
<br>
<input type="text">

// ✅ Correct
<img src="logo.png" />
<br />
<input type="text" />
⚠️ Rule 4: JavaScript expressions go inside {curly braces} 
// From your project's TeamMainContent.jsx:
<h2>{item.title}</h2>          // JavaScript variable
<span>{item.role}</span>             // Object property
<img src={item.image} />         // Dynamic attribute value

Fragments (<></>)

A Fragment lets you group a list of children without adding an extra DOM node (like a <div> wrapper).

// From your Home.jsx — using Fragment:
function Home() {
  return (
    <>                               // Fragment — no extra div in DOM
      <Header />
      <HomeMainContent />
      <Footer />
    </>
  )
}

// Same as:
import { Fragment } from 'react'
return (
  <Fragment>
    <Header />
    <HomeMainContent />
    <Footer />
  </Fragment>
)
💡 Why use Fragments? If you wrapped everything in a <div>, your DOM would have unnecessary nested divs. Fragments keep your HTML clean and semantic.
Chapter 4

React Components

What is a Component?

A component is a reusable piece of UI. Think of it as a JavaScript function that returns JSX. Every component in your project follows the same pattern.

// The simplest component pattern:
function ComponentName() {       // 1. Function (PascalCase)
  return (                        // 2. Return
    <div>                         // 3. JSX
      {/* your UI here */}
    </div>
  )
}

export default ComponentName     // 4. Export

Components in Your Project

Your project has three types of components:

TypeExamplesPurpose
Common ComponentsHeader, FooterShared across multiple pages
Page ComponentsHome, About, Services, Team, Contact, BlogDetailsOne per route — composes common + section components
Section ComponentsHomeMainContent, AboutMainContent, etc.Page-specific content rendered inside page components

Why Components Are Reusable

The same Header component is used on ALL 6 pages. If you need to change the navigation bar, you edit one file and all pages update instantly:

// Every page imports the same Header:
import Header from '../components/common/Header'

// Each page reuses it:
<Header />    // On Home page
<Header />    // On About page
<Header />    // On Services page — same component, same code, shared

Component Naming Rules

  • Must start with a capital letter — React distinguishes components from HTML tags by capitalization
  • Use PascalCase — HomeMainContent, BlogDetailsMainContent, ServicesMainContent
  • Filename usually matches component name — Header.jsx exports Header
🔑 Why Capital Letters? <div> is an HTML tag, but <Header /> is a React component. React uses the capitalization to decide: if it starts with lowercase → HTML element; if uppercase → React component.

Real Example from Your Project

// src/components/common/Header.jsx
function Header() {
  return (
    <nav className="navbar navbar-expand-lg">
      <div className="container">
        <a className="navbar-brand" href="index.html">
          IT<span>CO</span>
        </a>
        {/* ... navigation links ... */}
      </div>
    </nav>
  )
}

export default Header

Common Mistakes Beginners Make

❌ Lowercase component name: function header() { ... } → Won't work as a component. Always use function Header() { ... }.
❌ Forgetting to export: Creating a component without export default → Can't import it elsewhere.
❌ Component inside component: Defining a component inside another component → Causes re-creation on every render (performance bug).
Chapter 5

Import and Export

React uses ES Modules (ESM) — the modern JavaScript module system — to share code between files.

Default Export & Import

Used when a file exports one main thing:

// 📄 Exporting (e.g., Header.jsx)
function Header() {
  return <nav>...</nav>
}
export default Header   // Default export — one per file

// 📄 Importing (e.g., Home.jsx)
import Header from '../components/common/Header'
// Name can be anything: import MyHeader from './Header' works too!

Named Export & Import

Used when a file exports multiple things:

// 📄 Exporting
export const API_URL = 'https://api.example.com'
export function formatDate(date) { ... }

// 📄 Importing — must use EXACT names in curly braces
import { API_URL, formatDate } from '../utils/api'

// Can also rename with 'as':
import { formatDate as formatDisplayDate } from '../utils/api'

Relative Paths

Relative paths tell JavaScript where to find a file relative to the current file's location.

The Three Relative Path Symbols

SymbolMeaningExample
./Current directory./Header = "Header in this same folder"
../Parent directory (go up one level)../common/Header = "go up, then into common/"
../../Grandparent directory (go up two levels)../../assets/images/img.png
../../../Great-grandparent (go up three levels)Used when deeply nested

Visual Folder Diagram

📦 src ┣ 📂 components ┃ ┣ 📂 home ┃ ┃ ┗ 📜 HomeMainContent.jsx // ← We are HERE ┃ ┣ 📂 common ┃ ┃ ┣ 📜 Header.jsx // ../common/Header ┃ ┃ ┗ 📜 Footer.jsx // ../common/Footer ┃ ┣ 📂 services ┃ ┃ ┗ 📜 ServicesMainContent.jsx ┃ ┗ 📂 about ┃ ┗ 📜 AboutMainContent.jsx ┣ 📂 pages ┃ ┗ 📜 Home.jsx ┣ 📂 assets ┃ ┗ 📂 images ┃ ┗ 📜 img.png // ../../assets/images/img.png ┗ 📂 styles ┗ 📜 main.css

Notice that library imports like import Slider from 'react-slick' or import { BrowserRouter } from 'react-router-dom' do not use ./ or ../. These are bare imports — Vite looks them up in node_modules automatically.

Here is the complete rendering flow of your project, from the moment the browser loads the page to the final UI:

📄 index.html
↓

↓

Step-by-Step Flow

Step 1: index.html loads

The browser loads index.html which contains an empty <div id="root"> and a script tag pointing to src/main.jsx.

Step 2: main.jsx runs

// src/main.jsx — Your app's entry point
import React from 'react'
import ReactDOM from 'react-dom/client'
import { BrowserRouter } from 'react-router-dom'
import App from './App.jsx'

ReactDOM.createRoot(document.getElementById('root')).render(
  <BrowserRouter>
    <App />
  </BrowserRouter>
)

// 🔍 What's happening:
// 1. ReactDOM.createRoot() — creates a React root tied to <div id="root">
// 2. .render() — tells React: "Put this JSX tree inside the root"
// 3. <BrowserRouter> wraps App — enables routing everywhere

Step 3: App.jsx renders the current page

// src/App.jsx — Examines the URL, decides which page to show
import { Routes, Route } from 'react-router-dom'
import Home from './pages/Home'
import About from './pages/About'
// ... more page imports

function App() {
  return (
    <Routes>
      <Route path="/" element={<Home />} />
      <Route path="/about" element={<About />} />
      <Route path="/services" element={<Services />} />
      // ... more routes
    </Routes>
  )
}

Step 4: The page component renders its children

function Home() {
  return (
    <>
      <Header />            // Renders navigation bar
      <HomeMainContent />  // Renders hero section, services, testimonials
      <Footer />            // Renders footer with copyright
    </>
  )
}

Step 5: React reconciles and updates the DOM

React compares the new Virtual DOM tree with the previous one, calculates the minimal set of changes, and applies them to the real DOM. This process is called Reconciliation.

🔑 The Complete Rendering Chain:
Browser loads index.html → Vite injects main.jsx as a module → main.jsx renders <App /> inside <BrowserRouter> → App.jsx matches the URL → Renders the matching page (e.g., <Home />) → Home renders Header, HomeMainContent, Footer → Each component renders its JSX → React updates the real DOM → You see the page!

React Router

React Router is the standard library for navigation in React apps. It enables multi-page experiences inside a single-page application without full page reloads.

Your Project's Router Setup

// src/main.jsx — Router wraps the entire app
import { BrowserRouter } from 'react-router-dom'
import App from './App.jsx'

ReactDOM.createRoot(document.getElementById('root')).render(
  <BrowserRouter>        // Step 1: Wrap everything
    <App />
  </BrowserRouter>
)
// src/App.jsx — Route definitions
import { Routes, Route } from 'react-router-dom'
// ... page imports

function App() {
  return (
    <Routes>                          // Step 2: Match URL to routes
      <Route path="/" element={<Home />} />             // Home page
      <Route path="/about" element={<About />} />          // About page
      <Route path="/services" element={<Services />} />    // Services page
      <Route path="/team" element={<Team />} />          // Team page
      <Route path="/contact" element={<Contact />} />     // Contact page
      <Route path="/blog-details" element={<BlogDetails />} />
    </Routes>
  )
}

export default App

<Routes>Container for all route definitions; renders only the matching routeIn App.jsx

Route Explanation — Line by Line

<Route path="/services" element={<Services />} />
// ↑       ↑                ↑
// |       |                └── The component to render (in JSX format)
// |       └── The URL path to match (e.g., https://mysite.com/services)

How routing works:

  1. User clicks "Services" in the Header
  2. URL changes from / to /services
  3. <BrowserRouter> detects the URL change
  4. <Routes> checks each <Route>
  5. Finds the match: path="/services" matches the URL
  6. Renders <Services /> component
  7. Services component renders Header + ServicesMainContent + Footer
  8. Page content updates — no browser refresh!
💡 Best Practice: Replace all <a href="..."> tags in your Header with React Router's <Link> or <NavLink> to prevent full page reloads. <Link to="/services">Services</Link> is the React way.

Props

What are Props?

Props (short for "properties") are how data flows from a parent component to a child component. Props are read-only — a child component cannot modify its own props.

🔑 Key principle: In React, data flows DOWN the component tree. Parent → Child via props. Never the reverse (unless using callbacks or state).

Real Example: Services Component

Your ServicesMainContent renders multiple service cards from an array of objects. Each card receives data via props:

Common Props Pattern — Destructuring

When a component receives props, it's common to destructure them immediately for cleaner code:

// Without destructuring:
function ServiceCard(props) {

function ServiceCard({ title, description, icon }) {

Lists and Rendering

The map() Method

React uses JavaScript's map() to transform arrays into JSX elements. This is how your project dynamically renders the services, team members, testimonials, and blog posts.

// JavaScript .map() — converts each item into JSX
{servicesData.map((service, index) => (

The key Prop — Why It Matters

⚠️ Every item in a mapped list MUST have a unique key. React uses keys to identify which items changed, were added, or were removed. Never use the array index as a key if the list can change order. 
// ✅ Good — using a unique ID from your data
{items.map(item => <div key={item.id}>{item.title}</div>)}

// ⚠️ Acceptable for static lists — using index
{items.map((item, index) => <div key={index}>{item.title}</div>)}
💡 Best Practice: Your project uses key={service.id}, key={teamMember.id}, etc. — this is the correct approach! Always use a unique ID from your data when available.
Chapter 11

Conditional Rendering

Conditional rendering lets you show different UI based on conditions. React uses regular JavaScript control flows (if, ternary, &&) inside JSX.

Three Ways to Conditionally Render

1. Ternary Operator (condition ? A : B)

return (
  <div>
    {isLoggedIn ? (
      <button>Logout</button>
    ) : (
      <button>Login</button>
    )}
  </div>
)

2. && Operator (show/hide)

return (
  <div>
    {hasDiscount && (
      <span className="badge">50% OFF!</span>
    )}
    <p>Price: ${price}</p>
  </div>
)

// 🔍 How it works: If hasDiscount is true, React renders <span>50% OFF!</span>
// If hasDiscount is false, React renders nothing (false is ignored in JSX)

3. if/else Outside JSX

function Dashboard() {
  if (isLoading) {
    return <div>Loading...</div>      // Early return
  }

  if (error) {
    return <div>Error: {error.message}</div>
  }

Chapter 12

React Hooks

What are Hooks?

Hooks are functions that let you "hook into" React features from functional components. Before hooks (pre-React 16.8), you needed class components to use state and lifecycle methods. Hooks made React simpler and more powerful.

📜 History: Hooks were introduced in React 16.8 (February 2019). Before that, only class components could manage state. Now, functional components with hooks are the standard way to write React.

Rules of Hooks

  1. Only call hooks at the top level — not inside loops, conditions, or nested functions
  2. Only call hooks from React functions — functional components or custom hooks

useState — The Most Important Hook

useState lets you add state to functional components. It returns an array with two values: the current state and a function to update it.

import { useState } from 'react'

function Counter() {
  const [count, setCount] = useState(0)
  //       ↑        ↑              ↑
  //       |        |              └── Initial value (0)
  //       |        └── Setter function (call to update)
  //       └── Current state value (read only)

  return (
    <div>
      <p>You clicked {count} times</p>
      <button onClick={() => setCount(count + 1)}>
        Click me
      </button>
    </div>
  )
}

// 🔍 How it works:
// 1. First render: count = 0
// 2. User clicks button → setCount(0 + 1) → count = 1
// 3. React re-renders the component with the new count
// 4. DOM updates automatically — only the changed text!

useEffect — Side Effects

useEffect lets you perform side effects in your components — fetching data, subscribing to events, manipulating the DOM, setting timers.

import { useState, useEffect } from 'react'

function UserList() {
  const [users, setUsers] = useState([])

  useEffect(() => {
    // This runs AFTER the component renders
    fetch('/api/users')
      .then(res => res.json())
      .then(data => setUsers(data))
  }, [])   // Empty array = run only once (on mount)

  return (
    <ul>
      {users.map(user => (
        <li key={user.id}>{user.name}</li>
      ))}
    </ul>
  )
}

// 🔍 useEffect dependency array:
// useEffect(fn, [])      → Run once on mount
// useEffect(fn, [count]) → Run when 'count' changes
// useEffect(fn)          → Run on EVERY render (be careful!)
// useEffect(fn, []) with return → Cleanup on unmount

Real Project Example: Header Cleanup

Your Header.jsx uses useEffect with a cleanup function for the scroll event listener:

import { useState, useEffect } from 'react'

useEffect(() => {
  const handleScroll = () => {
    setScrolled(window.scrollY > 50)
  }
  window.addEventListener("scroll", handleScroll)

  // Cleanup function — runs when component unmounts
  return () => window.removeEventListener("scroll", handleScroll)
}, [])
💡 Why cleanup? If you add an event listener without removing it, and the component unmounts, the listener continues to exist in memory — causing a memory leak. The cleanup function prevents this.

Forms

Controlled Components

In React, form inputs whose value is controlled by useState are called controlled components. React, not the DOM, "owns" the input's value.

import { useState } from 'react'

function ContactForm() {
  const [formData, setFormData] = useState({
    name: '',
    email: '',
    message: ''
  })

  function handleChange(event) {
    const { name, value } = event.target
    setFormData(prev => ({ ...prev, [name]: value }))
  }

  function handleSubmit(event) {
    event.preventDefault()
    console.log(formData)  // { name: "John", email: "john@email.com", message: "..." }
  }

  return (
    <form onSubmit={handleSubmit}>
      <input
        name="name"
        value={formData.name}
        onChange={handleChange}
      />
      <input
        name="email"
        value={formData.email}
        onChange={handleChange}
      />
      <textarea
        name="message"
        value={formData.message}
        onChange={handleChange}
      />
      <button type="submit">Send</button>
    </form>
  )
}

Chapter 21

Interview Preparation

Beginner Level Questions

Q1: What is React and why was it created?
React is a JavaScript library for building user interfaces, created by Facebook in 2013. It was created to solve the problem of building large applications with data that changes over time. React introduced a component-based architecture and the Virtual DOM to make UI development more predictable and efficient.
Q2: What is JSX? How is it different from HTML?
JSX is a syntax extension that lets you write HTML-like code in JavaScript. Key differences: uses className instead of class, camelCase event names (onClick), self-closing tags are required (<br />), and JavaScript expressions are embedded with {}.
Q3: What is a React component?
A component is a reusable piece of UI that returns JSX. It's a JavaScript function (in modern React) that takes props as input and returns what should appear on the screen. Example: function Header() { return <nav>...</nav>; }
Q4: What is the Virtual DOM?
The Virtual DOM is a lightweight JavaScript representation of the real DOM. When state changes, React creates a new Virtual DOM tree, diffs it against the previous one, and updates only the changed elements in the real DOM. This is much faster than directly manipulating the real DOM.

Intermediate Level Questions

Q5: Explain the difference between state and props.
Props are read-only data passed from a parent component to a child. State is internal data managed by the component itself and can change over time. Props flow down (parent → child), state is local to its component. Changing state triggers a re-render; changing props (from parent) also triggers a re-render of the child.
Q6: What is React Router and how does it work?
React Router is a navigation library for React SPAs. It uses BrowserRouter to sync the UI with the URL. Routes components define URL-to-component mappings. When the URL changes, Router automatically renders the matching component without a full page reload. This enables fast, seamless navigation.
Q7: What is the purpose of the key prop in lists?
The key prop helps React identify which items in a list changed, were added, or removed. It should be a unique identifier from the data (like an ID). Using the array index as a key can cause bugs if the list order changes. Keys enable efficient re-rendering by allowing React to track each element.
Q8: Explain the rules of hooks.
Two rules: 1) Only call hooks at the top level of a component (not inside loops, conditions, or nested functions). 2) Only call hooks from React function components or custom hooks (not from regular JavaScript functions). Violating these can cause bugs where state gets out of sync.

Advanced Level Questions

Q9: How does React's reconciliation algorithm work?
When a component's state or props change, React creates a new Virtual DOM tree. The reconciliation algorithm compares (diffs) this new tree against the previous one. It uses two assumptions: different types of elements produce different trees, and stable elements with a key prop can be tracked across renders. React then computes the minimal set of DOM operations required and applies them in a batch.
Q10: What is the cleanup function in useEffect and when would you use it?
The cleanup function is returned from a useEffect callback and runs when the component unmounts or before the effect runs again. Common uses: removing event listeners (as in your Header's scroll listener), canceling API requests, clearing timers, and unsubscribing from WebSocket connections. Without cleanup, you can get memory leaks and stale data.
Q11: What is the difference between a controlled and uncontrolled component?
A controlled component has its value managed by React state — every state change updates the input's value, and every input change updates the state. An uncontrolled component manages its own internal state (using a ref to access the DOM directly). React recommends controlled components for most cases because they make the form data predictable and testable.
Q12: How would you optimize a React application that re-renders too often?
Techniques include: using React.memo to prevent unnecessary re-renders, using useMemo and useCallback to memoize expensive computations and functions, lifting state up or down appropriately, using the key prop correctly, code splitting with React.lazy, and profiling with React DevTools to identify bottlenecks.

Project-Specific Questions

Q13: Why is the Header component in a separate 'common' folder?
Because Header is used by multiple pages (Home, About, Services, Team, Contact, BlogDetails). Separating it into a reusable component prevents code duplication — changes to the navigation only need to be made in one file.
Q14: How does the testimonial slider work in this project?
It uses the react-slick library wrapped in a Slider component. Configuration is passed via a settings object: dots for navigation, autoplay for automatic sliding, speed for transition duration, and responsive breakpoints for mobile. Testimonial data is stored as an array and mapped into slide elements.
Q15: Why does the project use both Bootstrap classes and custom CSS?
Bootstrap provides the layout grid system, responsive breakpoints, and utility classes (margin, padding, etc.). Custom CSS (in main.css) handles brand-specific styling — colors, typography, animations, and design details that Bootstrap doesn't cover. This combines Bootstrap's speed with custom branding.
